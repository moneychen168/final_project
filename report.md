# Mini Chess AI 報告

## 1. 專案概述

本專案實作一個可以對弈 MiniChess（6×5 棋盤）的 AI，使用 Negamax 框架搭配多種搜尋優化技術，目標是在 10 秒時間限制內找出最佳棋步。

---

## 2. 狀態評估函數

評估函數定義在 `src/games/minichess/state.cpp` 的 `evaluate()` 中，從當前玩家的角度回傳分數，分數越高代表對自己越有利。

### 2.1 棋子分值

| 棋子 | 分值 |
|------|------|
| 兵（Pawn） | 20 |
| 車（Rook） | 60 |
| 馬（Knight） | 70 |
| 象（Bishop） | 80 |
| 后（Queen） | 200 |
| 王（King） | 1000 |

### 2.2 棋子位置加成（Piece-Square Table）

每種棋子在棋盤不同位置有不同的加分：
- **兵**：越靠近對方底線分數越高，鼓勵推進與升變
- **馬 / 象**：靠近中央加分，邊角扣分
- **王**：開局期間留在角落較安全，加分較高

### 2.3 國王威脅加成（King Tropism）

己方棋子距離敵方國王越近，加分越多，鼓勵 AI 主動包圍敵王、製造威脅。

$$加分 = 權重[棋子類型] \times (3 - 切比雪夫距離), \quad 距離 \leq 2$$

各棋子威脅權重：車=3、馬=3、象=2、后=5

### 2.4 兵的推進加成

每前進一格加 2 分，鼓勵兵的推進。

### 2.5 行動力加成（Mobility）

$$加分 = 2 \times （己方合法棋步數 - 對手合法棋步數）$$

行動力多代表棋形更活，選擇更多。

### 2.6 最終評估公式

$$分數 = 己方總分 - 對方總分 + 行動力加成$$

## 3. 搜尋演算法

### 3.1 Negamax（極小極大基礎框架）

使用 Negamax 框架統一雙方視角：子節點的分數取負值即為父節點的分數。

```
negamax(狀態, 深度):
    if 深度 == 0: return 評估(狀態)
    best = -無限大
    for 棋步 in 合法棋步:
        分數 = -negamax(下一狀態, 深度 - 1)
        best = max(best, 分數)
    return best
```

### 3.2 Alpha-Beta 剪枝

在 Negamax 基礎上加入 alpha-beta 剪枝。維護兩個邊界：
- **alpha**：目前已知己方最高分
- **beta**：對手會允許的最高分上限

當 `alpha >= beta` 時，對手不會走到這個分支，直接剪掉不用繼續搜尋，大幅減少節點數。

```
if alpha >= beta:
    break  # beta 剪枝
```

### 3.3 主要變化搜尋（PVS）

Alpha-Beta 的進一步優化。假設經過棋步排序後，第一個子節點通常是最佳棋步：

1. 第一個子節點用**完整窗口** `[-beta, -alpha]` 搜尋
2. 後續子節點先用**零窗口** `[-alpha-1, -alpha]` 快速驗證
3. 若零窗口搜尋 fail-high（表示可能更好），才重新用完整窗口搜尋

```
if 第一個子節點:
    分數 = -pvs(下一狀態, 深度-1, -beta, -alpha)
else:
    分數 = -pvs(下一狀態, 深度-1, -alpha-1, -alpha)  # 零窗口驗證
    if alpha < 分數 < beta:
        分數 = -pvs(下一狀態, 深度-1, -beta, -alpha)  # 重新完整搜尋
```

PVS 在大多數情況下可以跳過重搜，比純 Alpha-Beta 更快。

### 3.4 靜止搜尋（Quiescence Search）

在搜尋深度歸零時，若直接評估盤面可能因為「剛吃完子還未反吃」而高估/低估。靜止搜尋繼續展開所有**吃子**和**升變**棋步，直到盤面「安靜」再評估，解決 Horizon Effect 問題。

```
靜止搜尋(狀態, alpha, beta):
    stand_pat = 評估(狀態)
    if stand_pat >= beta: return beta
    alpha = max(alpha, stand_pat)
    for 棋步 in 吃子/升變棋步:
        分數 = -靜止搜尋(下一狀態, -beta, -alpha)
        if 分數 >= beta: return beta
        alpha = max(alpha, 分數)
    return alpha
```

預設靜止搜尋深度為 4。

## 4. 棋步排序

良好的棋步排序能讓 Alpha-Beta 剪枝效果最大化。排序優先順序如下：

| 優先級 | 類型 | 說明 |
|--------|------|------|
| 1 | 吃王 | 直接獲勝，最高優先 |
| 2 | MVV-LVA | 用低價值棋子吃高價值棋子優先 |
| 3 | 升變 | 兵升變為后 |
| 4 | 中央控制 | 目標格靠近棋盤中心加分 |

---

## 5. 反覆加深搜尋（Iterative Deepening）

由 UBGI 協定層控制，從深度 1 開始依序搜尋到時間用完。每完成一層都輸出當前最佳棋步，確保時間到期時一定有答案可以輸出。

---

## 6. 重複局面偵測

使用 **Zobrist Hashing** 對每個盤面產生 64-bit 雜湊值，並透過 GameHistory 記錄歷史局面出現次數。當同一局面出現 3 次或以上，判定為平局，避免重複循環。
